% ===== ABSTRACT =====
\begin{abstract}
Code generation nowadays is producing executable code from natural language descriptions. This paper explores the latest neural approaches to code generation, focusing on how models understand natural language and synthesize structured programs. This paper reviews three papers: CodeT5~\cite{wang-etal-2021-codet5}, InCoder~\cite{fried2023incodergenerativemodelcode}, and StructCoder~\cite{10.1145/3636430}. Here I am discussing the methodologies, strengths, and limitations of each approach, and how they contribute to advancing the state of the art in code generation. Here CodeT5 presents an identifier-aware encoder-decoder architecture for code understanding and generation~\cite{wang-etal-2021-codet5}. InCoder focuses on code infilling and synthesis through a generative modeling approach~\cite{fried2023incodergenerativemodelcode}. StructCoder adds structure awareness to improve generation quality on code translation and text-to-code tasks~\cite{10.1145/3636430}.
\end{abstract}

\section{Introduction}
The idea of code generation from natural language has been fascinating researchers for a long time. With the progressive work in deep learning and some transformer-based models, we have seen significant advancements in this area. The ability of machines to understand natural language and generate structured code has the potential to revolutionize software development. But the issue has been that while models can generate code, they often struggle with understanding the gist of programming logic and structure, which eventually leads to code that may be syntactically correct but semantically flawed. But sometimes even syntactical correctness is flawed as the models are not even aware of the identifiers and structure of the code. This is where models like CodeT5, InCoder, and StructCoder come into play, each addressing different aspects of the code generation problem.

These three models have introduced three different mechanisms to improve code generation. CodeT5 is built on the T5 architecture which is a transformer-based model that has been adapted for code understanding and generation. But on top of it, CodeT5 incorporated the idea of being identifier-aware, which means that it can better understand the role of identifiers in code, which is crucial for generating meaningful and functional code. InCoder, on the other hand, greatly focuses on code infilling and synthesis. It uses a generative modeling approach which allows it to fill in missing parts of code and generate new code based on the context provided. This is particularly useful for tasks like code completion and code repair. StructCoder went one more step beyond. It added structure awareness to the model, which means that it can better understand the hierarchical and logical structure of the code. All three models have shown significant improvements in code generation, but they also have their own limitations.

\section{Literature Review}
\subsection{Background and Problem Formulation}
When we talk about code generation, we essentially mean two things: (1) \textit{natural language to code} (NL-to-PL) generation, where the model takes a natural language description and generates code. (2) \textit{code to natural language} (PL-to-NL) generation, where the model takes code and generates a natural language description of what the code does. Both of these tasks are important for different aspects. The former is crucial for automating software development, where developers can simply describe what they want in natural language and the model can interpret that and generate the corresponding code. The latter is important for code documentation and understanding, where developers use natural language to explain what code does. It helps in understanding the codebase and maintaining the codebase effectively. It eventually helps in documentation and code review processes. But both tasks need to understand the intent, generate syntactically correct code, and also generate semantically meaningful code. This is where the challenge lies.
Early code generation focused on neural networks that were trained on large code corpora. But the transformer's self-attention mechanism proved to be much more of a game changer for capturing long-range dependencies in code~\cite{vaswani2023attentionneed}. Pre-training on large unlabeled code corpora (e.g., GitHub) became standard, with objectives such as masked language modeling (MLM), denoising autoencoding (DAE), and replaced token detection (RTD).
\subsection{Key Challenges in Neural Code Generation}
Based on the three papers, we can identify several key challenges in natural code generation.
\begin{itemize}
	\item \textbf{Understanding Natural Language}: One of the big challenges out there is that the model needs the ability to understand natural language descriptions accurately. But the thing is that natural language can be ambiguous. The model needs to understand the intent behind the natural language description.
	\item \textbf{Identifier Semantics}: As highlighted by CodeT5, understanding identifiers and their semantics is crucial for generating meaningful code. Identifiers are the names given to variables, functions, classes, etc. If the model does not understand the semantics of identifiers, it may generate code that might be syntactically correct but semantically meaningless. 
	\item \textbf{Structural Awareness}: From StructCoder~\cite{10.1145/3636430}, we can see that if we understand the structure of the code, then we can generate better code. Code has a hierarchical structure, and understanding this structure is crucial for generating code that is not only syntactically correct but also semantically meaningful. Here the models need to understand how the different parts of the code relate to each other and how they fit together.
	\item \textbf{Infilling and Editing}: InCoder~\cite{fried2023incodergenerativemodelcode} focuses on code infilling and synthesis. This is a huge challenge as the models need to be able to fill in missing parts of code and generate new code based on the context provided. This feature is particularly useful for tasks like code completion and code repair, but it requires the model to have a deep understanding of the code context and also the ability to generate code that fits well within that existing context.
	\item \textbf{Bidirectional Alignment}: Another challenge is bidirectional alignment. This means that the model should be well aligned in both directions: from natural language to code and from code to natural language.
\end{itemize}
\subsection{Related Pre-trained Models for Code}
	Several pre-trained models have been proposed other than the three focal papers I would be discussing. CodeBERT~\cite{feng2020codebertpretrainedmodelprogramming} is an encoder-only model trained with MLM and replaced token detection on NL-PL pairs. GraphCodeBERT~\cite{guo2021graphcodebertpretrainingcoderepresentations} incorporates data flow graphs into the encoder. PLBART~\cite{ahmad2021unifiedpretrainingprogramunderstanding} is an encoder-decoder model pre-trained with denoising autoencoding on code and natural language. Even though these models are somewhat effective, the decoder here lacks awareness of code structure.
\section{Methodologies}
\subsection{CodeT5: Identifier-aware Encoder-Decoder Model}
CodeT5 is built on the T5 architecture which is a transformer-based model which has been used for various natural language processing tasks  (e.g, Code Understanding, code generation, code translation, etc.). Firstly if I want to summarize the architecture of CodeT5 in two words then it would be Explicit and Focused understanding. CodeT5 is designed to be identifier-aware, which means that it understands the role of identifiers in code generally in a better way than other compatriots.
For natural language processing tasks, there were several powerful models like BERT, GPT, T5, etc. These models were pre-trained on large corpora of text data and then for smaller tasks they were fine tuned. As a result, even for less amount of data, there was good performance. So researchers were thinking that if we could do the same thing for code, then we could also get good performance for code generation and code understanding tasks. So they were trying to apply NLP models. Example:
\begin{lstlisting}
def add(a, b):
    return a + b
\end{lstlisting}
Then the model can learn that it is a function and then perform addition. But the main problem occurred that the model seemed to be one-sided. Some models like BERT were just better for understanding the code, and some models acted like just decoders (e.g. GPT), which were just better for generating code. But we wanted both generation and understanding. So these single-type models are not enough for us. The second problem was that the previous models were just taking the words as a sequence. But in reality, code is much more structured. Example:
\begin{lstlisting}
binarySearch(arr.left,right,x)
\end{lstlisting}
Here the binary search is a function name and those arr, left, right, x are variables. So in CodeT5, they proposed a unified model where the encoder and decoder are trained together. And the main idea around CodeT5 is:
\begin{itemize}
	\item \textbf{Identifier-aware Learning}: The model would understand which one is an identifier and which one is not. Example:
	\begin{lstlisting}
def binarySearch(arr.left,right,x)
	\end{lstlisting}
	Here the model would understand that binarySearch is a function name and arr is an array and left, right are position variables. Understanding these meanings is necessary.
	\item \textbf{Trained using Masked Language Modeling}: 
	It means that when training, some identifiers are masked and the model is trained to predict those masked identifiers. This helps the model to learn the semantics of identifiers and how they relate to each other in the code.
	\item \textbf{Using code and comment together}: For a lot of code there are comments. Example:
	\begin{lstlisting}
# increment value
def inc(x):
	return x + 1
	\end{lstlisting}
	Here the model would learn comment-to-code relation. So if we give the input phrase ``increment value'', then the model would understand that it needs to generate a function that increments a value. The vice versa is also true.
	\item \textbf{Dual learning}: The model is trained in a dual learning manner, which means it is trained for NL-to-PL generation and PL-to-NL generation at the same time.
\end{itemize}
The previous models would see the code as a sequence of tokens.
Code snippet:
\begin{lstlisting}
def add(a, b):
	return a + b
\end{lstlisting}
Here the previous models would just see it as a sequence of tokens such as \texttt{def}, \texttt{add}, \texttt{a}, and \texttt{b}, along with punctuation symbols. But the problem around that was that all models being treated as the same. But in code all tokens are not equal. Here \texttt{def} is a keyword and \texttt{add}, \texttt{a}, and \texttt{b} are identifiers. The identifiers being so much important are because with those developer gives their code a meaning and other people then would understand what the code is trying to interpret and deliver.

The issues of previous models are:
\begin{itemize}
	\item \textbf{Identifier Recognition Issue}: Previously, models would not recognize function names and array names. They would just see them as some random words.
	\item \textbf{Lost of Meaning}: Code snippet:
	\begin{lstlisting}
def binarySearch(arr.left,right,x)
	\end{lstlisting}
	Before, the model would think that binarySearch here is just a token. But in reality, it is a search algorithm, which they would not understand.
	\item \textbf{Problem in Masking}: Code Snippet:
	\begin{lstlisting}
def Mask(arr, left, right, x):
	\end{lstlisting}
	The previous models would just guess randomly as they would not know that they are identifiers.
\end{itemize}
Then in CodeT5:
\begin{itemize}
	\item \textbf{Detecting Identifier}: Here it is explicitly trained to detect identifiers. So it would understand that binarySearch is a function name and arr is an array and left, right are position variables. Understanding these meanings is necessary.
	\item \textbf{Specialized Training}:
		\begin{itemize}
			\item \textbf{Task 1: Identifier Tagging}:
			\begin{itemize}
				\item \texttt{arr} $\rightarrow$ 1 (identifier)
				\item \texttt{def} $\rightarrow$ 0 (not identifier)
			\end{itemize}
			\item \textbf{Task 2: Masked Identifier Prediction}:
			Code Snippet:
			\begin{lstlisting}
def Mask(arr, left, right, x):
			\end{lstlisting}
			The model is trained to predict masked identifiers. So it would learn the semantics of identifiers and how they relate to each other in the code. So this model would have to recover binarySearch from the masked code snippet.
		\end{itemize}
\end{itemize}
The reason behind CodeT5 using T5 architecture is that using one model here, code summarization, generation, translation, and defect detection can be done. So it is a unified model for all code related tasks. The main idea here is that the encoder and decoder are trained together. So the model can learn to understand the code and also generate the code at the same time. T5 uses an Encoder-Decoder structure. There, the encoder understands the input (code/text) and the decoder generates the output. But in CodeT5, it not only understands the code but also generates the code and understands the identifiers. So CodeT5 added identifier awareness and special training objectives to the T5 architecture to improve code understanding and generation performance.
The main ideas of pretraining in CodeT5 are:
\begin{itemize}
	\item \textbf{Identifier-aware Masked Span Prediction}: Example 1:
	\begin{lstlisting}
def add(a, b):
	return a + b
	\end{lstlisting}
	After Masking the code snippet:
	\begin{lstlisting}
def [MASK](a, b):
	return a + b
	\end{lstlisting}
	Here identifiers are masked more and more as the identifiers are meaningful and important. 
	Example 2:
	\begin{lstlisting}
def calculateTotal(price):
	\end{lstlisting}
	After Masking the code snippet:
	\begin{lstlisting}
def <mask>(<mask>):
	\end{lstlisting}

	Here the model would guess calculateTotal and price. So the model gets compelled to learn the context and semantics.

	\item \textbf{Bimodal Dual Generation}:
	Here the model is trained in two ways:
	\begin{itemize}
		\item \textbf{(A) Natural language to code generation}
		\textbf{Input (Example):}
		\begin{lstlisting}
Add two numbers
		\end{lstlisting}
		\textbf{Output:}
		\begin{lstlisting}
def add(a, b):
    return a + b
		\end{lstlisting}

		\item \textbf{(B) Code to natural language generation}
		\textbf{Input (Example):}
		\begin{lstlisting}
def add(a, b):
    return a + b
		\end{lstlisting}
		\textbf{Output:}
		\begin{lstlisting}
Add two numbers
		\end{lstlisting}
	\end{itemize}
	As a result, the model understands the code and also it can write the code better.
	\item \textbf{Multi-task Learning}:
	Here all tasks are trained together. As a result, the model becomes much more versatile and can perform all the tasks using one model.


\end{itemize}

The architecture of CodeT5 is based on the T5 model, which is an encoder-decoder transformer architecture. The encoder takes the input and decoder generates the output. The main novelty here is that the model is trained to be identifier-aware.

\subsection{InCoder: Generative Model for Code Infilling and Synthesis}

InCoder mainly introduces a generative modeling approach for code infilling and synthesis. The main idea is that the model can fill in missing parts of code and generate new code based on the context provided. For previous models like GPT/Codex, it would just generate code from left to right. But in InCoder, it can generate code in any direction. Now I will discuss the training idea here:
Suppose the original code snippet is:
	\begin{lstlisting}
def add(a, b):
	return a + b
\end{lstlisting}
\begin{itemize}
	\item \textbf{Step 1: Masking a little portion}:
	\begin{lstlisting}
def add(a, b):
	<MASK>
	\end{lstlisting}
	\item \textbf{Step 2: Taking the hidden part at last}:
	\begin{lstlisting}
def add(a, b):
	<MASK>
<MASK> return a + b <EOM>
	\end{lstlisting}
	Now the model is trained here to predict the sequence. As a result, the model learns that there is code at the top, at the bottom there is context, and then what would be the code in the middle. So it learns to fill in the missing part. It can be generalized as such that Left context + Right context $\rightarrow$ Missing part. So the model learns to generate code based on the context provided. This is bidirectional context and it is very useful for code infilling and synthesis. 
	
\end{itemize} 
Inference in Real:
Suppose the code snippet is:
\begin{lstlisting}
def square(x):
	<MASK>
\end{lstlisting}
Now we give it to the model. Then the model can predict the missing part. And the model gives the output as:
\begin{lstlisting}
	return x * x
\end{lstlisting}

Real use cases according to the paper~\cite{fried2023incodergenerativemodelcode} are:
\begin{itemize}
	\item \textbf{Variable Rename}:
	Before:
	\begin{lstlisting}
x = 4
y = x + 2

	\end{lstlisting}
	Masked code:
	\begin{lstlisting}
<MASK> = 4
y = <MASK> + 2
	\end{lstlisting}
	Model:
	\begin{lstlisting}
num = 4
y = num + 2
	\end{lstlisting}
	\item \textbf{Docstring Generation}:
	\begin{lstlisting}
def add(a, b):
	return a + b
	\end{lstlisting}
	Masked code:
	\begin{lstlisting}
def add(a, b):
	"""<MASK>"""
	return a + b

	\end{lstlisting}
	\item \textbf{Type Prediction}:
	\begin{lstlisting}
def add(a: int, b: int) -> <MASK>:
	return a + b
	\end{lstlisting}
	The output would be:
	\begin{lstlisting}
	int
	\end{lstlisting}
	\item \textbf{Missing Code Fill}:
	\begin{lstlisting}
for i in range(n):
	<MASK>
	\end{lstlisting}
	The output would be:
	\begin{lstlisting}
print(i)
	\end{lstlisting}

\end{itemize}
The reason this model is this much powerful is that it can understand the context from both sides. So then it is able to generate code that fits very well within the existing context. For old models, it was:
only Left $\rightarrow$ Right.
But in InCoder, it is left context + right context $\rightarrow$ Middle Fill. So it is much more powerful and useful for code infilling and synthesis.
The intuitive idea here is that suppose we are solving a problem and writing code:
 \begin{lstlisting}
	for(int i = 0; i < n; i++){
		//something missing
	}
 \end{lstlisting}
InCoder basically sees the problem, loop, and context and then tries to guess what should be inside. 
Suppose we have a code snippet:
\begin{lstlisting}
def add(a, b):
	return a + b
\end{lstlisting}

Suppose we are trying to mask:
\begin{itemize}
\item Left context: \texttt{def add (a, b):}
\item Right context: \texttt{`` ''}
\item Span = \texttt{return a + b}
\end{itemize}

\noindent\textbf{Training Objective~\cite{fried2023incodergenerativemodelcode}}
\begin{equation}
\log P\left([
\text{Left};\langle \text{Mask:0} \rangle;
\text{Right};\langle \text{Mask:0} \rangle;
\text{Span};\langle \text{EOM} \rangle
]\right)
\end{equation}

Code snippet:
\begin{lstlisting}
def add(a, b):
	<MASK>
<MASK> return a + b <EOM>
\end{lstlisting}
Here first \verb|<MASK>| means that something is missing. Then the second \verb|<MASK>| means to generate the code in the middle. And lastly \verb|<EOM>| means that the code is ended. So the model here is learning that there is code at both left and right and trying to predict the missing span. It is kind of learning substring reconstruction problem.

\noindent\textbf{Inference (Real Use)~\cite{fried2023incodergenerativemodelcode}}
\begin{equation}
P\left(\cdot \mid [
\text{Left};\langle \text{Mask:0} \rangle;
\text{Right};\langle \text{Mask:0} \rangle
]\right)
\end{equation}
Code snippet:
\begin{lstlisting}
def add(a, b):
	<MASK>
\end{lstlisting}
Model internally takes it as Left + Mask + Right + Mask
Then it generates the code in the middle and gives the output as:
\begin{lstlisting}
	return a + b	
\end{lstlisting}
Full Flow:
Training time:
Original code $\rightarrow$ mask $\rightarrow$ move span to end $\rightarrow$ train
Inference time:
Give code with \verb|<MASK>| $\rightarrow$ model generates the missing part

Example (Multi-region):
\begin{lstlisting}
def process(x):
	<MASK1>
	y = x * 2
	<MASK2>
\end{lstlisting}
Here the model can fill:
\begin{lstlisting}
<MASK1> print(x)
<MASK2> return y
\end{lstlisting}

In the previous models:
\begin{equation}
P(\text{token} \mid \text{previous tokens})
\end{equation}
In InCoder:
\begin{equation}
P(\text{token} \mid \text{left context} + \text{right context})
\end{equation}
It basically means that it is not sequence to sequence. Rather it is autoregressive model plus smart training trick. So it means that it is not architectural change. Only the training objective is changed here.
All GPT-based models learn like~\cite{fried2023incodergenerativemodelcode}:
\begin{equation}
P(x_1, x_2, x_3, \ldots, x_n) = \prod_{i=1}^{n} P(x_i \mid x_1 \ldots x_{i-1})
\end{equation}
Here every token learns to predict from the before tokens.
Example:
\begin{lstlisting}
def max(a, b):
	if a > b:
		return a
	else:
		return b
\end{lstlisting}
Now:
\begin{lstlisting}
def max(a, b):
	<MASK>
	if a > b:
		return a
	else:
		return b
\end{lstlisting}
Here the model would have the context of \texttt{if a > b:} and it does not know \texttt{else: return b}. So it may guess \texttt{return a}. But it is just a lucky guess, not properly context based. But in InCoder, it would have those contexts and it would be able to generate the code in the middle with much more accuracy. 
\subsubsection{Model Architecture Overview}
The GPT-based architecture model used to see the left context and it would not be able to fill in the middle with much more accuracy. But in InCoder, these things are fixed more or less.
\begin{itemize}
	\item \textbf{Masking the Random Span}
	\item \textbf{Moving the Masked Span to the End}
	\item \textbf{Training the Model to Predict the Masked Span}
\end{itemize}
\subsection{StructCoder: Structure-aware Code Generation}
In the transformer-based models, the main problem is that they used to see the code as text. But in reality, the code has its syntax and also the semantics (meaning).
\begin{lstlisting}
a = 5
b = a + 2
\end{lstlisting}
Here the syntax has to be alright and also the semantics here would be that b is dependent on a. So StructCoder says that code is just not a natural language. Rather the code is a combination of Structure, Dataflow and Syntax. They mainly used two things:
\begin{itemize}
	\item \textbf{Abstract Syntax Tree (AST)}: It is the structure tree of the code.
	\begin{lstlisting}
		a = b + c
	\end{lstlisting}
	\begin{center}
	\begin{forest}
	for tree={
		draw,
		rounded corners,
		minimum height=9mm,
		minimum width=9mm,
		inner sep=2pt,
		s sep=10mm,
		l sep=10mm,
		font=\ttfamily\bfseries
	}
	[{$=$}
		[a]
		[+
			[b]
			[c]
		]
	]
	\end{forest}
	\end{center}
	It says which one is operator and which one is variable.
	
	\item \textbf{Dataflow Graph}: 
	It says which variable is taking value from which variable. Example:
	\begin{lstlisting}
a = 5
b = a + 2
	\end{lstlisting}
Here the dataflow graph would be:
a $\rightarrow$ b
\end{itemize}
StructCoder Architecture:
StructCoder = Encoder + Decoder (Transformer-based)
\begin{itemize}
	\item \textbf{Encoder}: It can understand the input. The StructCoder encoder uses AST and DFG and a special attention mechanism to understand the structure of the code. It can understand the syntax and semantics of the code.
	\item \textbf{Decoder}: Decoder generates output. Before the decoder used to just predict the next token. But in StructCoder, the decoder is given two extra tasks:
	\item \textbf{AST Path Prediction}: The model is trained to predict the output code's AST path. So it learns the structure of the code.
	\begin{lstlisting}
		a = b + c
	\end{lstlisting}
	Here the model doesn't just predict the code, it also predicts the AST path. 
	\item \textbf{Dataflow Prediction}: Model learns which variable depends on which one.
	\begin{lstlisting}
		b = a + 2
	\end{lstlisting}
	The model can understand a $\rightarrow$ b

\end{itemize}

\noindent\textbf{Pre-training Strategy:}
StructCoder uses Denoising Autoencoder (DAE) as the pre-training strategy. What it does:
\noindent Code + AST + DFG $\rightarrow$ it is corrupted.\\
\noindent Then it is asked to the model to recover the original code.
Example:
\begin{lstlisting}
	a = b + c
\end{lstlisting}
After Corruption:
\begin{lstlisting}
	a = __ + c
\end{lstlisting}
Then the model would fill it with b.
\subsubsection{Structure-Aware Encoder}

The structure-aware encoder treats code as more than a simple token sequence. For plain text input, the model can rely only on the surface tokens. When we take code input, it uses the code tokens together with structural information from the AST and DFG.\@ This design follows StructCoder~\cite{10.1145/3636430} and here it allows the code not only to take the lexical form of the program but also to understand the syntax and semantics through the AST and DFG representations. 

\paragraph{Input representation}
For a code example, the encoder input can be written as follows, following StructCoder~\cite{10.1145/3636430}:
\begin{equation}
\mathbf{x} = [\mathrm{CLS}, s_1, s_2, \ldots, s_{|S|}, \mathrm{SEP}, l_1, l_2, \ldots, l_{|L|}, v_1, v_2, \ldots, v_{|V|}],
\end{equation}
where $s_i$ denotes the code tokens, $l_i$ denotes AST leaves, and $v_i$ denotes variables from the DFG.\@ In practice, this means that the encoder can combine token-level information with tree-level and graph-level structure.

For example, the statement \texttt{a = b + c} can be represented by the token sequence for the code itself, while the AST and DFG provide additional structure. In a text-only scenario, the model can ignore AST and DFG signals. But in a code scenario, the full structural input is used.

\paragraph{Input embeddings}
All three input types are mapped into the same vector space. Code tokens use a standard embedding lookup, DFG variables are assigned variable-specific embeddings, and AST leaf embeddings are built from the path between the root and the leaf. StructCoder~\cite{10.1145/3636430} defines the leaf embedding for a leaf node $l$ as
\begin{equation}
E(l) = \sum_{i=1}^{|l|} E_{\mathrm{type}}(r_i.\mathrm{type}) \odot E_{\mathrm{height}}(|l|-i),
\end{equation}
where $r_i$ is the $i$-th node on the path from the root to the leaf. The type embedding captures the node label, while the height embedding helps the model understand the position of the node inside the tree. As a result, the model can learn to recognize patterns in the AST structure.

\paragraph{Structure-aware self-attention}
The encoder then applies different attention patterns based on the type of relationship being modeled, again following StructCoder~\cite{10.1145/3636430}.

\begin{itemize}
	\item \textbf{Code-to-code attention}: this is the standard token-to-token attention used in transformer models, with relative position information added to preserve order, as described in StructCoder~\cite{10.1145/3636430}.
	\begin{equation}
	A(s_i, s_j) = E_{s_i}^{\top} W_q^{\top} W_k E_{s_j} + \phi(|i-j|).
	\end{equation}
	Meaning:
	Standard T5 attention. The relative position function $\phi$ helps the model understand the order of tokens, which is crucial for code where the sequence matters.


	\item \textbf{Leaf-to-leaf attention}: this attention compares two AST leaves by how much tree context they share. Two leaves that lie on a longer common path are treated as more closely related in StructCoder~\cite{10.1145/3636430}.
	\begin{equation}
	\operatorname{sim}(l_i, l_j) = \log\!\left(1 + \frac{\sum_{k=1}^{\min(|l_i|,|l_j|)} \mathbb{1}(r_k^i = r_k^j)}{|l_i||l_j|}\right).
	\end{equation}
	The final attention score combines the learned similarity with the normal dot-product attention:
	\begin{equation}
	A(l_i, l_j) = E_{l_i}^{\top} W_q^{\top} W_k E_{l_j} + \bigl(w_a \cdot \operatorname{sim}(l_i, l_j) + w_b\bigr).
	\end{equation}
	So here it is not based on the position rather is based on the similarity.
	Example:
	\begin{lstlisting}
		= -> + -> b
		= -> + -> c
	\end{lstlisting}
	Here the two leaves \texttt{b} and \texttt{c} share a common path through the AST, so they would have higher attention between them than two leaves that do not share as much structure.

	\item \textbf{Variable-to-variable attention}: in the DFG, attention is allowed only between variables that are connected by a dependency edge, following StructCoder~\cite{10.1145/3636430}.
	\begin{equation}
	A(v_i, v_j) =
	\begin{cases}
	E_{v_i}^{\top} W_q^{\top} W_k E_{v_j}, & D_{ij} = 1, \\
	-\infty, & \text{otherwise}.
	\end{cases}
	\end{equation}
	Meaning:
	Only connected variables get the attention here. 
	Example:
	\begin{lstlisting}
		b = a + 2
	\end{lstlisting}
	Allowed: 
	\begin{lstlisting}
		a <-> b
	\end{lstlisting}
	Not Allowed: 
	\begin{lstlisting}
		a <-> unrelated variable
	\end{lstlisting}

	\item \textbf{Code-to-leaf and code-to-variable attention}: these cross-type links connect code tokens with the AST leaves and DFG variables that correspond to them, following StructCoder~\cite{10.1145/3636430}.
	\begin{equation}
	A(s_i, l_j) =
	\begin{cases}
	E_{s_i}^{\top} W_q^{\top} W_k E_{l_j}, & L^{ast}_{ij} = 1, \\
	-\infty, & \text{otherwise}.
	\end{cases}
	\end{equation}
\end{itemize}
Meaning:
token $\rightarrow$ leaf: if the token corresponds to an AST leaf, then it can attend to that leaf.
Example:
\begin{lstlisting}
	a = b + c
\end{lstlisting}
Links:
\begin{lstlisting}
	a <-> leaf(a)
	b <-> leaf(b)
\end{lstlisting}

This design gives the encoder a more complete view of the program.\@ Rather than treating the code as a flat sequence of tokens, the model can understand the hierarchical structure of the code through the AST and the dataflow through the DFG.\@ With this attention mechanism, the model can learn to recognize patterns in the code structure.
\subsubsection{Structure-Aware Decoder}

\begin{figure}[H]
\centering
\includegraphics[width=0.9\textwidth]{structdecoder.png}
\caption{Structure-aware decoder generates the next token in the target code and predicts the node types on the root-to-leaf path to the leaf containing this token in the target AST as well as the DFG edges incident on this token~\cite{10.1145/3636430}.}\label{fig:structcoder_decoder}
\end{figure}
In a normal decoder, it just predicts the next token. But in StructCoder, the decoder does predict the next token, AST path, and also DFG relations. So it learns the syntax and semantics together.

The interesting thing here is that the output token is not increasing here, rather it is doing extra tasks which helps it in training time. So in inference time, it gets more awareness and prediction becomes better.

\paragraph{Three training tasks}
StructCoder trains the decoder with three complementary objectives:
\begin{itemize}
	\item \textbf{LM}: next-token prediction.
	\item \textbf{APP}: AST path prediction.
	\item \textbf{DFP}: dataflow prediction.
\end{itemize}

The language modeling head is the familiar decoder objective. For a hidden state $h_i$, the probability of the next token is computed as StructCoder~\cite{10.1145/3636430}:
\begin{equation}
p_i = \operatorname{softmax}(W h_i).
\end{equation}
This is the usual transformer-style prediction step: the model takes the hidden representation and turns it into a distribution over the vocabulary. The corresponding loss is also defined in StructCoder~\cite{10.1145/3636430}:
\begin{equation}
L_{lm} = -\frac{1}{|T|} \sum_{i=1}^{|T|} \log p_i(t_i).
\end{equation}
This part is actually same as the standard language modeling. So it just ensures that the model learns to generate the correct token sequence overall.

The AST path prediction head is what makes the decoder syntax-aware. For every token, it predicts root$\rightarrow$ leaf path. \@ If we take the example \texttt{return a + b}, then the token \texttt{b} is not just a token by itself; it also sits somewhere in the tree, for example along a path like \texttt{=} $\rightarrow$ \texttt{+} $\rightarrow$ \texttt{b}. The decoder learns to recover that path. Formally, the path-level prediction is written as StructCoder~\cite{10.1145/3636430}:
\begin{equation}
p^{ast}_{ik} = \operatorname{softmax}\!\left(W^{( |l_i|-k )}_{ast} h_i\right).
\end{equation}
Here the model predicts the $k$-th node on the path for token $i$. The loss for this task is defined in StructCoder~\cite{10.1145/3636430}:
\begin{equation}
L_{app} = -\frac{1}{|T|} \sum_{i=1}^{|T|} \sum_{k=1}^{|l_i|} \log p^{ast}_{ik}(r_k^i.\mathrm{type}).
\end{equation}
In intuitive words, here the model learns where to place which token in tree. So it results in syntax-aware generation.

The third task is dataflow prediction. Here the model gets to understand the semantic relationships. For every pair of tokens $(i, j)$, the decoder predicts if there is a dataflow link from token $j$ to token $i$. For example, in \texttt{a = b + c}, both \texttt{b} and \texttt{c} influence \texttt{a}. So the model should learn that \texttt{b $\rightarrow$ a} and \texttt{c $\rightarrow$ a}. StructCoder models this relation as
\begin{equation}
p^{dfg}_{ij} = \sigma\!\left(h_i^{\top} U_{dfg}^{\top} V_{dfg} h_j + u_{dfg}^{\top} h_i + v_{dfg}^{\top} h_j + w_{dfg}\right).
\end{equation}
The sigmoid makes the result a probability, so the model can decide whether the edge exists or not. It also implies that direction matters. The corresponding binary cross-entropy loss is also from StructCoder~\cite{10.1145/3636430}:
\begin{equation}
L_{dfp} = -\frac{1}{|T|^2} \sum_{i=1}^{|T|} \sum_{j=1}^{|T|} \Big[y_{ij} \log p^{dfg}_{ij} + (1-y_{ij}) \log(1-p^{dfg}_{ij})\Big].
\end{equation}
It eventually helps in semantic-aware generation.

\paragraph{Final objective}
StructCoder combines the three losses into one final objective~\cite{10.1145/3636430}:
\begin{equation}
L = L_{lm} + \alpha_1 L_{app} + \alpha_2 L_{dfp}.
\end{equation}
Here $L_{lm}$ controls token correctness, $L_{app}$ controls syntax correctness, and $L_{dfp}$ controls semantic correctness. The weights $\alpha_1$ and $\alpha_2$ balance how much the syntax and dataflow tasks matter during training. So the decoder is still generating code normally, but it learns from AST and DFG supervision at the same time~\cite{10.1145/3636430}.

For example, if the model sees \texttt{a = b + c}, the language modeling head learns to generate the token sequence, the AST path head learns the tree structure of each token, and the DFG head learns that \texttt{b} and \texttt{c} flow into \texttt{a}. It is the core idea behind this Structure-aware decoder.

\subsection{Strengths and Weaknesses}

\begin{itemize}
	\item \textbf{CodeT5}: 
	CodeT5's main strength is that using one model it can simultaneously understand and generate code. It is more of like performing multiple tasks with one model. It is trained to be identifier-aware and it is specifically trained to understand which one is variable and which one is function name. But it has its fair share of weaknesses. It does not use AST or DFG.\@ So it does not understand the syntax structure of the code and does not even understand the dataflow of the code. So it faces problems in generating complex code.
	\item \textbf{InCoder}: 
	InCoder's main strength is that it uses a novel training objective which is code infilling using bidirectional context. When normal models used to see only left context, InCoder sees both left and right context. So it is much more powerful for code infilling and synthesis. But it has some drawbacks too. The decoder of InCoder doesn't understand any structure. It has even no structural inductive bias. So even if syntax is correct, it may generate code which is not semantically correct.
	\item \textbf{StructCoder}: StructCoder's main strength is that it has explicitly incorporated it to be structure-aware. It uses AST and DFG to understand the structure of the code. So it is more syntax and semantic aware. AST path prediction helps it to understand syntax structure and Data Flow Prediction helps it to understand the variable flow. So ultimately the code becomes more syntactically and semantically correct. But its weakness is its preprocessing. So the sequence overhead becomes bigger.
\end{itemize}
\section{Discussion}
CodeT5 mainly focused on identifier (variable and function name) awareness. It trains for the purpose of identifying variable and keyword. As a result, understanding code becomes easier. Whereas for the case of InCoder, it tries to understand the code by filling in the missing part. So it is more of a training strategy. It is not an architectural change. Rather it is a training change in a smart way. And then StructCoder is more of an architectural change. It uses AST and DFG to understand the structure of the code. So it is more of a structural change. Every model has its good and bad aspects. The problem with CodeT5 is that it does not use AST/structure. So it does feel the issue in generating complex code. For the case of InCoder, it doesn't understand the syntax or dataflow. So it may generate code which is not syntactically correct or semantically correct. But for the case of StructCoder, it is more aware of the structure and semantics. So it generates better code. But the issue with this is that as we add AST and DFG to it, so the input size goes bigger and inference becomes slower. So it is a trade-off between performance and speed.
\section{Limitations and Future Proposed Idea}
I would like to combine those three models together. I would like to do InCoder-style infilling where it can fill in the missing part after watching the left and right context. Rather than using full AST input, I would like to work on graph transformer-based architecture where it can understand the structure of the code without having to input the full AST.\@ So it would be more efficient and faster.\@ It was likely the limitation of StructCoder that it has to input the full AST and DFG.\@ And I would also like to add CodeT5-style identifier-aware training objectives to it. So it would be more of a unified model which can understand the code better and also generate better code in a faster way.

\section{Conclusion}
In this paper, I have analyzed three influential code generation models: CodeT5, InCoder, and StructCoder. CodeT5 here introduces a unified encoder-decoder architecture with identifier-aware pre-training tasks, improving code understanding and generation. Whereas InCoder focuses on a novel training objective which models code infilling with bidirectional context, enabling it to generate code that fits well within existing code. And StructCoder incorporates the idea of checking if it is syntactically correct. So I think unified work would be really interesting where we can combine the best of all three worlds. Then it may achieve a higher benchmark performance and also be more efficient in terms of inference time.
